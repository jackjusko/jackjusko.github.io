# Voice and editing rules

## Input

The user provides a brain dump — spoken notes, bullet points, rough paragraphs, or stream-of-consciousness. Treat it as the source of truth for **intent** and **claimed experience**.

## Output voice

- **Always first person** (I, my, we only when genuinely collaborative)
- Preserve the user’s register:
  - Casual dumps → conversational but structured
  - Formal dumps → polished essay tone, still first person
- Do not turn posts into third-person bios or press releases

## Structural editing (encouraged)

- Add a clear title and lede
- Break into H2 sections with descriptive headings
- Cut repetition, filler, and false starts
- Reorder for narrative flow (setup → what happened → what I learned)
- Fix grammar, punctuation, and unclear pronouns

## Factual editing (careful)

**Do fix when:**
- Obvious typo in a tool name, date, or URL
- Internal contradiction within the dump
- Claim conflicts with portfolio facts (employer names, project names, degree)

**Do not:**
- Invent projects, credentials, or outcomes the user did not claim
- Add statistics or quotes without source in the dump
- Upgrade casual speculation into definitive claims

**Flag in confirmation** when you made non-obvious factual corrections or removed unsupported claims.

## Anti-AI prose

Before delivery, read the user’s global `first-person-enthusiast-writer/ai-patterns.md` if available:

`~/.cursor/skills/first-person-enthusiast-writer/ai-patterns.md`

Banned patterns include: contrast frames (“not X, but Y”), AI vocabulary clusters, cinematic compression, puffery, vague authorities.

## Tags

Infer 2–5 tags from content. Prefer existing tags on the site when they fit. Tags are topical labels, not a category system.

## Length

No fixed word count. Match the dump: short notes stay short; deep dives can run long. Every post still gets full SEO head tags and home funnel blocks.
